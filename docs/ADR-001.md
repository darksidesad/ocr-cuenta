# ADR-001: Uso de OpenRouter con Gemini Flash para extracción de facturas

## Estado
Aceptado

## Contexto
El proyecto requiere un servicio de LLM para extraer datos estructurados de texto de facturas colombianas. El LLM debe:
- Entender español colombiano y formato DIAN
- Retornar JSON válido y consistente
- Ser costo-efectivo para procesamiento de documentos
- Estar disponible vía API HTTP

## Decisión
Usar **OpenRouter** como gateway con el modelo **google/gemini-2.0-flash-001** como LLM principal.

## Alternativas consideradas

| Alternativa | Pros | Contras | Decisión |
|-------------|------|---------|----------|
| OpenAI (GPT-4o) | Alta calidad, structured outputs | Costo alto ($5/1M tokens input) | Rechazado |
| OpenAI (GPT-4o-mini) | Buen balance costo/calidad | Menor calidad con documentos largos | Rechazado |
| Anthropic (Claude) | Excelente razonamiento | Costo alto, rate limits estrictos | Rechazado |
| Google directo (Gemini) | Mismo modelo, sin intermediary | Requiere billing Google Cloud | Rechazado |
| OpenRouter + Gemini Flash | Barato ($0.075/1M tokens), structured outputs, sin lock-in | Depende de disponibilidad OpenRouter | **Seleccionado** |
| Local (Ollama) | Sin costo API, sin dependencia externa | Calidad inconsistente, lento en CPU, sin structured outputs nativos | Rechazado para v1 |

## Consecuencias

### Positivas
- Costo extremadamente bajo (~$0.0001 por factura procesada)
- Sin lock-in: OpenRouter permite cambiar modelo con 1 línea de código
- Structured outputs nativos (response_format: json_object)
- Latencia aceptable (< 5s para documentos típicos)

### Negativos
- Dependencia de servicio externo (OpenRouter + Google)
- Rate limits posibles en tier gratuito
- Calidad variable según complejidad del PDF

### Mitigaciones
- Retry con backoff exponencial
- Reintentar 1 vez si JSON inválido
- Fallback: cambiar modelo en .env sin modificar código
- Logging de cada extracción para debugging
